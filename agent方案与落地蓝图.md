# 体系文件维护 Agent 方案与落地蓝图

## 1. 方案定位

你前面的 `skill.md` 已经定义了这个能力要做什么；如果要真正落地，推荐不要只做成一个“万能 Agent”，而是拆成 **1 个编排 Agent + 4 个执行 Agent + 1 个审计模块**。

这样做的原因很简单：
- 检索和写回是两类完全不同的能力。
- Word / Excel / PDF 的定位逻辑不同，不能只靠一次字符串替换。
- 逐条确认属于强交互流程，适合独立控制。
- 审计日志和回滚最好单独留一层，避免流程耦合过重。

---

## 2. 推荐架构

### 2.1 总体结构

```text
用户
  ↓
Orchestrator Agent（总控编排）
  ├─ Retrieval Agent（全局检索）
  ├─ Locate Agent（精确定位）
  ├─ Review Agent（生成审阅卡片 + 等待确认）
  ├─ Writeback Agent（安全写回 + 备份 + 回滚）
  └─ Audit Logger（审计日志 / 修订报告）
```

### 2.2 各 Agent 职责

#### A. Orchestrator Agent
负责整条链路的调度。

主要职责：
1. 接收用户输入：旧语句、新语句、范围、文件类型、匹配模式。
2. 调用 Retrieval Agent 生成候选命中。
3. 调用 Locate Agent 补足章节、页码、Sheet、单元格等位置信息。
4. 将每个命中送给 Review Agent 逐条审阅。
5. 根据用户确认结果决定是否调用 Writeback Agent。
6. 最后汇总结果并交给 Audit Logger 输出报告。

#### B. Retrieval Agent
负责“找全”。

主要职责：
1. 对 `Target_Sentence` 做精确匹配。
2. 对关键词拆分后做召回。
3. 对同义改写做语义检索。
4. 合并、去重、排序，输出候选命中列表。

建议输出字段：
- `file_name`
- `file_path`
- `snippet`
- `score`
- `source_type`
- `raw_hit_id`

#### C. Locate Agent
负责“找准”。

主要职责：
1. 对候选命中进行二次解析。
2. 在原文档里定位其真实位置。
3. 提取上下文和结构节点。
4. 判断该位置是否适合自动替换。

建议输出字段：
- `file_name`
- `file_path`
- `doc_type`
- `section`
- `paragraph_index`
- `page_number`
- `sheet_name`
- `cell`
- `matched_text`
- `context_before`
- `context_after`
- `replace_risk`

#### D. Review Agent
负责人机交互。

主要职责：
1. 生成可读的审阅卡片。
2. 展示 Before / After。
3. 提示风险。
4. 接收用户指令：确认、跳过、编辑、终止。

推荐交互指令：
- `1`：确认修改
- `2`：跳过本条
- `3`：手动编辑本条替换文本
- `4`：终止本轮任务
- `5`：本文件后续相同命中全部套用当前决定

#### E. Writeback Agent
负责“安全改”。

主要职责：
1. 检查文件是否在白名单目录。
2. 创建备份。
3. 按文件类型调用对应写回器。
4. 校验写回结果。
5. 失败时回滚。

#### F. Audit Logger
负责“留痕”。

主要职责：
1. 记录每一条命中的审阅与执行结果。
2. 记录备份路径、修改时间、操作者。
3. 输出修订报告。
4. 支持按任务编号回查。

## 3. 为什么不建议只做一个 Agent

如果全部塞进一个 Agent，通常会遇到四个问题：

1. **检索不稳定**
   - 既要找全又要找准，很容易把语义近似和真实可替换命中混在一起。

2. **交互状态容易乱**
   - 一边检索、一边确认、一边写回，状态机会非常混乱。

3. **文档格式差异太大**
   - `.docx`、`.xlsx`、`.pdf` 的处理逻辑根本不是一套。

4. **审计难做**
   - 一旦修改出错，很难复盘到底是在检索阶段错了，还是写回阶段错了。

所以更推荐：
- 大脑只有一个：Orchestrator
- 手脚拆开：检索、定位、审阅、写回、审计分别处理

## 4. 推荐状态机

```text
INIT
  ↓
SEARCHING
  ↓
LOCATING
  ↓
REVIEWING
  ├─ confirm → WRITING
  ├─ skip → REVIEWING(next)
  ├─ edit → WRITING
  └─ quit → TERMINATED
WRITING
  ├─ success → REVIEWING(next)
  ├─ fail → ERROR / RETRY / ROLLBACK
  └─ all done → REPORTING
REPORTING
  ↓
DONE
```

建议每一条命中都生成唯一 ID，例如：
`TASK20260415-003-HIT07`

这样后面审计、回滚、错误追踪都会方便很多。

## 5. 文件类型处理策略

### 5.1 Word / docx
推荐定位方式：
- 文档标题
- 章节号
- 段落索引
- 表格编号 + 行列号

推荐写回方式：
- 使用结构化解析器读取段落和表格
- 不建议把 docx 转纯文本后再盲替换

### 5.2 Excel / xlsx
推荐定位方式：
- Sheet 名
- 单元格地址
- 表头字段
- 命中值所在行

推荐写回方式：
- 基于单元格级别更新
- 保留原格式、公式、批注、合并关系

### 5.3 PDF
推荐策略：
- PDF 更适合作为“检索与定位来源”
- 不适合作为主要写回载体

建议：
- 若 PDF 来源是 Word 导出的，尽量改 Word 源文件
- 若只有扫描版 PDF，只给用户展示候选，不自动改写

### 5.4 Markdown / TXT
最适合自动化。

推荐写回方式：
- 结构定位 + 精确替换
- 写回后做二次校验

## 6. 检索策略建议

### 第一层：精确检索
适合：
- 用户明确给出完整句子
- 希望高精度命中

### 第二层：关键词拆分检索
适合：
- 一句话在不同文件里有轻微变体
- 某些标点、空格、单位写法不同

### 第三层：语义检索
适合：
- 同义改写
- 近似表述
- 被拆成两句的同一规则

### 第四层：规则过滤
用于过滤这类不应直接修改的命中：
- 目录
- 页眉页脚
- 修订记录
- 表头
- 文件编号
- 版本号

## 7. 审阅卡片推荐格式

```markdown
### 命中 7 / 23
- 文件：来料检验管理程序.docx
- 路径：/docs/iqc/来料检验管理程序.docx
- 位置：4.2 节，第 3 段
- 风险：低

#### 上下文
……仓库收料完成后，所有来料须在 24 小时内完成检验，并形成记录……

#### 修改前
所有来料须在 24 小时内完成检验

#### 修改后
所有来料须在 12 小时内完成检验

#### 操作
回复：
1 = 确认修改
2 = 跳过本条
3 = 手动编辑
4 = 终止任务
```

## 8. 写回前必须做的 6 个检查

1. 该文件路径是否在白名单目录中。
2. 本条命中是否已有用户确认。
3. 待替换文本是否仍然存在，避免因文件被他人改动导致错位。
4. 备份是否成功生成。
5. 当前命中位置是否唯一。
6. 替换后文本是否通过二次校验。

二次校验建议：
- 替换后重新读取该位置
- 确认新文本存在
- 确认旧文本已消失或只在允许区域残留

## 9. 你这类场景最适合的落地形态

### 方案 A：Dify Chatflow
适合：
- 想先快速搭出可演示版本
- 对“逐条确认”要求高
- 文件写回通过外部 API 实现

推荐分层：
- Chatflow 负责交互
- 外部服务负责检索索引与文件写回

### 方案 B：Coze 工作流
适合：
- 更偏 Bot 交互
- 已经有插件体系
- 需要卡片式确认

### 方案 C：自建 Web + Agent Backend
最适合正式上线。

推荐结构：
- 前端：任务创建页 + 命中审阅页 + 修订报告页
- 后端：检索服务 + 文档解析服务 + 写回服务 + 审计服务
- Agent：只负责判断和编排，不直接碰底层文件系统

## 10. 最推荐的真实落地方案

如果你是准备真正投入使用，而不只是做 Demo，我建议：

### 最佳实践组合
- **前端**：Web 审阅台
- **编排层**：1 个 Orchestrator Agent
- **检索层**：全文检索 + 向量检索
- **定位层**：按格式解析器处理 docx / xlsx / pdf
- **写回层**：独立文件写入 API
- **审计层**：数据库记录任务、命中、确认、写回结果、备份路径

### 原因
因为你这个场景本质上不是“聊天问答”，而是：
**文档治理 + 人工审核 + 批量修订 + 合规追踪**。

这类系统最终都要走向：
- 可回放
- 可追责
- 可回滚
- 可审计

单靠一个提示词很难长期稳定。

## 12. 给你的最终建议

如果你现在要开始做，建议按下面顺序推进：

### 第一阶段：先做 MVP
先支持：
- `.docx`
- `.xlsx`
- 精确检索
- 逐条确认
- 备份 + 写回 + 报告

### 第二阶段：再加增强能力
再加：
- 模糊检索
- 语义近似召回
- 相同命中的批量授权
- 风险分级

### 第三阶段：再做正式版
最后补：
- 任务中心
- 修订历史
- 一键回滚
- 权限分级
- 多人协同审阅

---

## 13. 一句话总结

这类需求最合适的方案不是“一个大 Agent 直接改文件”，而是：

**编排 Agent + 检索 Agent + 定位 Agent + 审阅 Agent + 写回 Agent + 审计模块**

其中：
- Agent 负责判断和交互
- 外部服务负责文档解析、写回和留痕
- 用户始终保留逐条确认权

这样才能既提效，又不把体系文件改乱。
